# Gap Filling (LEAD / LAG)

> [!IMPORTANT]
> **「状態が変わった時しかログが飛んでこない」**
> IoTセンサー、ステータス変更履歴、CRMの営業フェーズ移行ログなど、世の中の多くのデータは「差分」で保存されます。
> これを時系列のタイムラインとして「面」で捉える（欠損値を前後の値で埋める）技術は、データエンジニアリングの最難関の一つです。

---

## 1. ビジネスシナリオ：状態のギャップ埋め

「IoTセンサーからデバイスの稼働状態が送られてくる。通信量を抑えるため、変化があった瞬間（Activeになった、Errorになった）しかデータが来ない。」
ある毎分・毎時の状態を見たい場合、データが存在しない時間は**「直前のステータスを継続している」**とみなして穴埋めする必要があります。

**【インプット：生ログ（差分）】**

| Device | Time | Status |
|---|---|---|
| D1 | 10:00 | Active |
| D1 | 10:05 | Error |
| D1 | 10:15 | Active |

**【目的：1分ごとのマスターと結合した時の理想結果】**

| Device | Time | Status (本来の姿) |
|---|---|---|
| D1 | 10:00 | Active |
| D1 | 10:01 | **Active (前を継承)** |
| D1 | 10:02 | **Active (前を継承)** |
| ... | ... | ... |
| D1 | 10:05 | Error |

---

## 2. 解決策：LAST_VALUE + IGNORE NULLS

標準的なSQLでこの「前方への穴埋め（Forward Fill）」を行おうとすると、複雑な再帰CTEや自己結合が必要になります。しかし、モダンSQLでは `IGNORE NULLS` を用いて一撃で解決可能です。

### ✅ 実装パターン：前方穴埋め

前提として、すべてのタイムスタンプ（毎分）の箱だけを持つカレンダーテーブルとLEFT JOINした結果、`Status`がNULLだらけになったテーブルを想像してください。

```sql
SELECT
  device,
  time,
  status AS original_status,
  
  -- ★ 直前までの行から、一番最後の（＝最新の直近の）生きた値を取得する
  LAST_VALUE(status) IGNORE NULLS OVER (
    PARTITION BY device         -- デバイスごとに独立して見る
    ORDER BY time ASC           -- 時間を昇順に並べる
    -- ★ 窓枠：先頭から「現在の行」まで
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS forward_filled_status

FROM device_status_timeline
```

**【この構文の仕組み】**
`IGNORE NULLS` は、Window枠内の値を探す際、値がNULLの行を完全にスキップします。現在行のステータスがNULLの場合、システムは上（過去）に向かって遡り、最初に見つけた「NULLではない実体値」を採用します。

## 3. 次のイベントを予測する (LEAD / LAG)

ギャップ埋めと並んで重要なのが、「このイベントが起きてから、次のイベントが起きるまでの**滞在時間（リードタイム）**を知りたい」という要件です。

### 3.1 ❌ 自己結合を使う（アンチパターン）
「今の行」の次のデータを探すために、`t1.time < t2.time` などを条件に自己結合すると、O(N^2)の計算量となり一瞬でサーバーが落ちます。

### 3.2 ✅ LEAD による次行へのアクセス
`LEAD()` 関数は、物理的に「次の行」の値を瞬時に取得します。

```sql
SELECT
  user_id,
  event_time,
  event_type,
  
  -- 1. 次に起きたイベントの時間を持ってくる
  LEAD(event_time) OVER (
      PARTITION BY user_id 
      ORDER BY event_time ASC
  ) as next_event_time,
  
  -- 2. DATEDIFFで時間差分を計算（例：秒単位）
  datediff(
    second, 
    event_time, 
    LEAD(event_time) OVER (PARTITION BY user_id ORDER BY event_time ASC)
  ) as duration_seconds

FROM user_events
```

> [!TIP]
> `LEAD` の逆で「前の行」の値を取得するのは `LAG` 関数です。
> `LEAD(column, 1)` は1個次、`LEAD(column, 3)` なら3個次の値を取るなど、ステップ数の指定も可能です。さらに、次の行が存在しない（最後の行の）場合に返すデフォルト値も `LEAD(column, 1, 'End')` のように設定できます。

## 4. 確認クイズ（理解度チェック）

### Q1. LEADとLAGの違いは？

<details>
<summary>💡 回答を見る</summary>

**正解:** `ORDER BY` で指定されたソート順に対して、
- **LAG**: 前の行（過去）の値を取得する。
- **LEAD**: 後の行（未来・次）の値を取得する。
</details>

### Q2. IGNORE NULLS は何をしてくれる機能ですか？

<details>
<summary>💡 回答を見る</summary>

**正解:** ウィンドウ関数（LAST_VALUEなど）が評価対象の範囲（窓）を探す際、値がNULLであるレコードを「そもそも存在しないもの」として無視し、NULL以外の有効な値を確実に拾い上げる手助けをしてくれます。前回の状態を引き継ぐ Forward Fill の必須機能です。
</details>
