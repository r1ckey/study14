# 2.3: 配列データの展開（EXPLODE）

---

### 1. 【エンジニアの定義】Professional Definition

> **Nested Data (ネスト構造データ)**:
> JSONやParquet形式でよく見られる、1つのカラムの中に「配列(Array)」や「構造体(Struct)」が含まれている状態。
> 
> **EXPLODE()**:
> 1つのカラムに含まれる配列の要素を、独立した「行」に分解（爆発）させるテーブル生成関数。非正規化データを正規化形式（1行1事実）に戻す際に不可欠。

---

### 2. 【0ベース・深掘り解説】Gap Filling

#### 🧐 なぜ「横」ではなく「縦」にするのか？
ログデータには `purchased_items: ["Apple", "Orange", "Banana"]` のように、購入商品が配列で入っていることがあります。
このままだと「Appleが何個売れたか」を集計するには、文字列検索（LIKEなど）が必要になり、非常に低速で不正確です。

集計の鉄則は **「1つのセルには 1つの値だけを入れる」** こと。 `EXPLODE` を使って、3つのアイテムを 3行に分解することで、通常の `COUNT` や `GROUP BY` が使えるようになります。

### 4. 【技術実装】Implementation Best Practices

#### ✅ 配列の縦展開
```sql
SELECT 
  order_id,
  user_id,
  item_name
FROM silver.orders
-- 配列を縦に展開し、item_name というエイリアスを付ける
LATERAL VIEW EXPLODE(item_list) AS item_name;

-- Databricks SQL のモダンな書き方
-- SELECT order_id, EXPLODE(item_list) FROM silver.orders;
```

#### ✅ 出現順序（位置）を維持する (POSEXPLODE)
配列の「何番目にあったか」に意味がある場合（例：検索結果の順位など）に使用します。
```sql
SELECT
  search_id,
  rank + 1 AS search_rank, -- インデックスは0から始まるため+1
  content_id
FROM silver.search_logs
-- pos (position) と content_id の両方を取り出す
LATERAL VIEW POSEXPLODE(results_array) AS rank, content_id;
```

---

### 5. 【Key Takeaways】

- **集計前の縦展開**: 配列内の要素を分析対象（GROUP BYのキーなど）にしたい場合は、まず `EXPLODE` で縦持ちに変換する。
- **LATERAL VIEW**: 既存の列（order_idなど）を維持しつつ新しい列を生成する場合の標準的な構文。
- **重複に注意**: 1行を複数行に増やすため、展開後に集計を行う場合は、分母の数（Order数など）が狂わないように `COUNT(DISTINCT order_id)` を使う等の配慮が必要。
