# Handling Duplicates (ROW_NUMBER)

> [!CAUTION]
> **「テーブル結合（JOIN）のテストをしたら、なぜか件数が1.5倍に増えていた」**
> 実務で最もよく遭遇し、かつ最も危険なバグの温床が「一意（ユニーク）だと思っていたIDの重複」です。
> この重複が引き起こす JOIN 爆発（Cross Join状態）を防ぐための、絶対的防御策を解説します。

---

## 1. ビジネスシナリオ：重複データの存在

「ユーザーの住所マスタがある。複数回引っ越した人もいるが、売上データには**常にユーザーの最新の住所**だけを紐付けたい。」

**【インプットデータ：住所履歴（address_history）】**

| user_id | address | update_date |
|---|---|---|
| U100 | 東京 | 2024-01-01 |
| U100 | 神奈川 | 2024-06-15 |

このまま売上テーブルと `user_id` だけで JOIN すると、売上の行が「過去の住所」「今の住所」の2行に分裂・複製されてしまいます。

---

## 2. ❌ アンチパターン (MAXとGROUP BY)

初心者がやりがちなのが、`MAX(date)` などを使って一旦最新日を出し、それを自己結合（Self JOIN）で元のテーブルに戻す方法です。

```sql
-- ❌ コードが長く、二度読みが発生する遅いクエリ
SELECT a.user_id, a.address, a.update_date 
FROM address_history a
INNER JOIN (
    -- 最新の日付を特定する
    SELECT user_id, MAX(update_date) as max_date 
    FROM address_history 
    GROUP BY user_id
) b
-- IDと最新日の両方で結合して元のレコードを引っ張り出す
ON a.user_id = b.user_id 
AND a.update_date = b.max_date
```

**【何が危険なのか？】**
もし、全く同じ日に2回住所変更のレコードが入っていた場合（例：オペレーターのシステム入力ミス）、`update_date = max_date` となる行が**2行**存在してしまうため、**結局重複は排除できません。**

---

## 3. ✅ ベストプラクティス：ROW_NUMBER関数

データの「グループ（PARTITION）」ごとに、「並び順（ORDER）」を決めて背番号を振り、常に【1番】だけを抜き出す方式です。これなら同日重複があっても強制的に1件に絞り込めます。

### 3.1 CTEを利用した標準実装（汎用的）

```sql
WITH numbered_history AS (
  SELECT 
    user_id,
    address,
    update_date,
    -- ROW_NUMBER でグループごとに連番を振る
    ROW_NUMBER() OVER (
      PARTITION BY user_id      -- 誰ごとにグループ化するか？
      ORDER BY update_date DESC -- 新しい順（降順）に1, 2, 3...
    ) as row_num
  FROM address_history
)
-- 必ず「1」のものだけを抽出する
SELECT user_id, address, update_date 
FROM numbered_history 
WHERE row_num = 1;
```

### 3.2 同日重複を制御する「Tie-Breaker」のテクニック
もし同日に2件レコードが存在した場合、システムはランダムで1番を選びます。安全性を高めるため、`ORDER BY` に**決定打となる第2のキー（IDや更新時間など）**を入れます。

```sql
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY update_date DESC, system_record_id DESC -- ★第2のキー
    ) as row_num
```

---

## 4. 🥇 プロの最速技：QUALIFY句 (Databricks / Snowflake)

モダンなDWH（Databricks, Snowflake, BigQueryなど）で標準化されているのが `QUALIFY` 句です。
`WHERE` が集計前の行を絞り、`HAVING` が集計後の行を絞るなら、**`QUALIFY` はWindow関数の結果で集計後の行を絞ります。**
CTEすら不要になります。

```sql
-- ✅ 究極にシンプルな1文
SELECT 
  user_id, 
  address, 
  update_date 
FROM address_history
-- Window関数の結果が「1」である行だけを残す
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY user_id 
  ORDER BY update_date DESC
) = 1;
```

> [!TIP]
> **ROW_NUMBER vs RANK vs DENSE_RANK**
> 重複を【強制的に1行だけ残す】排除目的であれば、何があっても連続する連番を振る **ROW_NUMBER()** が正解です。
> `RANK()` を使うと、同日レコードの順位が `1, 1, 3...` となり、結局結合時に1位が2行抽出されてしまい、JOIN爆発のバグを引き起こします。

## 5. 確認クイズ（理解度チェック）

### Q1. 以下の要件を満たす場合、PARTITION BY と ORDER BY には何を指定すべきか？
「"category"ごとに、一番価格（price）が安い商品を抽出したい」

<details>
<summary>💡 回答を見る</summary>

```sql
ROW_NUMBER() OVER (PARTITION BY category ORDER BY price ASC)
```
の計算結果が `1` のものを抽出します。
</details>
