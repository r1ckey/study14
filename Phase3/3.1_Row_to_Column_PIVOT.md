# Row to Column (PIVOT)

> [!IMPORTANT]
> **「テーブルの特定の文字列を、カラム名（列名）に昇格させたい」**
> システムはデータを保存する際、拡張しやすいように「縦待ち（正規化）」で保存します。しかし、BIツールや機械学習モデルは「横持ち」の特徴量テーブルを要求します。
> この縦から横へのトランスフォームを自在に操れるかどうかが、データエンジニアの腕の見せ所です。

---

## 1. ビジネスシナリオ：アンケートデータの横持ち化

「ユーザーからのアンケート回答がデータベースに蓄積されている。質問ごとの回答が縦に入っているが、これを機械学習の特徴量（1ユーザー1行）に変換したい。」

**【インプット（縦持ち：EAVモデル）】**

| user_id | question_id | answer |
|---|---|---|
| U1 | Q1 | Yes |
| U1 | Q2 | No |
| U2 | Q1 | No |

**【理想のアウトプット（横持ちの特徴量）】**

| user_id | answer_q1 | answer_q2 | answer_q3 |
|---|---|---|---|
| U1 | Yes | No | NULL |
| U2 | No | NULL | NULL |

---

## 2. ❌ アンチパターン：無限の自己結合 (Self-Join)

SQLの基礎機能だけでこれを作ろうとすると、質問の数だけ `LEFT JOIN` を繰り返すハメになります。

```sql
-- ❌ コードが肥大化し、結合するたびにパフォーマンスが極端に悪化する
SELECT 
  u.user_id,
  q1.answer AS answer_q1,
  q2.answer AS answer_q2
FROM (SELECT DISTINCT user_id FROM survey_results) u
LEFT JOIN survey_results q1 ON u.user_id = q1.user_id AND q1.question_id = 'Q1'
LEFT JOIN survey_results q2 ON u.user_id = q2.user_id AND q2.question_id = 'Q2'
-- 質問が100個あったら終わる...
```

---

## 3. ✅ 解決策1：CASE WHEN + GROUP BY（古典的だが最強）

全てのRDBMSで確実に動き、パフォーマンスも安定している伝統的かつ実践的な手法です。

```sql
-- ✅ 実務で最も愛される確実な手法
SELECT 
  user_id,
  -- Q1の答えだけを取り出し、それ以外は無視して統合する
  MAX(CASE WHEN question_id = 'Q1' THEN answer END) AS answer_q1,
  MAX(CASE WHEN question_id = 'Q2' THEN answer END) AS answer_q2,
  MAX(CASE WHEN question_id = 'Q3' THEN answer END) AS answer_q3
FROM survey_results
GROUP BY user_id;
```

**【なぜ `MAX` を被せるのか？】**
`GROUP BY user_id` をした瞬間、DBエンジンは「U1」の行を1行にまとめようとします。文字列であっても、「Q1のYesか、それ以外のNULLか」を比較して生き残らせるために、ダミーの集計関数として `MAX`（または `MIN`）を通すのがセオリーです。

---

## 4. ✅ 解決策2：PIVOT 構文（モダンな書き方）

モダンなSQLエンジンでは `PIVOT` キーワードによる専用構文が用意されています。

```sql
SELECT *
FROM (
  -- ① PIVOTさせたい元のテーブル定義
  SELECT user_id, question_id, answer 
  FROM survey_results
)
PIVOT (
  -- ② 交差点（セル）に入れたい値の【集計関数】
  MAX(answer) 
  
  -- ③ 【元の値】を【新しいカラム名】に変換するマッピング
  FOR question_id IN (
    'Q1' AS answer_q1, 
    'Q2' AS answer_q2, 
    'Q3' AS answer_q3
  )
)
```

\`\`\`mermaid
table
    User_ID --> PIVOT_Engine
    Question --> PIVOT_Engine
    Answer --> PIVOT_Engine
    
    PIVOT_Engine --> Col_U1[U1 | Yes | No]
    PIVOT_Engine --> Col_U2[U2 | No | NULL]
\`\`\`

## 5. 確認クイズ（理解度チェック）

### Q1. PIVOTを使う際、集計関数（MAXやSUM）を書かずにそのまま値を入れることは可能ですか？

<details>
<summary>💡 回答を見る</summary>

**正解: 不可能です。**

PIVOTの裏側では必ず `GROUP BY` に相当する集計が走ります。なぜなら「U1がQ1を2回回答していた場合」データベースはどうしていいか分からないためです。文字なら `MAX`、数値なら `SUM` や `AVG` など、意図を示す関数が必須です。
</details>

### Q2. 実行するまで対象の質問の数や種類（Q1, Q2, Q4...）が分からない場合、動的にカラム名を作る Dynamic PIVOT は通常のSQLで書けますか？

<details>
<summary>💡 回答を見る</summary>

**正解: 直接は書けません。**

標準SQLはコンパイル時に出力カラムのスキーマが確定している必要があります。動的にカラム数が増えるPIVOTを行いたい場合は、Python（PySparkの `df.pivot()`）を使うか、SQL生成のストアドプロシージャを組む必要があります。
</details>
