# Difference Extraction (EXCEPT / ANTI JOIN)

> [!CAUTION]
> **「テーブルAにあって、テーブルBにないデータを探したい」**
> まさか `NOT IN (SELECT ...)` を使っていませんか？数万件以上のデータでこれをやると、クエリが永遠に返ってこなくなります。
> 大規模分散処理において最も安全で最速の『差分抽出設計』をマスターしましょう。

---

## 1. ビジネスシナリオ：新規・退会・差分の特定

「毎日数百万件のユーザーマスタが外部DBから送られてくる（全量スナップショット）。前日のリストと本日のリストを比較して、『新しく追加されたID（新規入会）』だけにウェルカムメールを送りたい。」

**【今月の全ユーザー (A)】**：100万人
**【先月の全ユーザー (B)】**：98万人

---

## 2. ❌ アンチパターン：NOT IN サブクエリ

もっとも「人間の言葉（自然言語）」に近いですが、SQLエンジンにとっては悪夢のような重さを誇る書き方です。

```sql
-- ❌ テーブルAの全ての行に対し、テーブルBの中にないかフルスキャンを行おうとする場合がある
SELECT user_id, email 
FROM users_this_month
WHERE user_id NOT IN (
    SELECT user_id FROM users_last_month
);
```

**【致命的な問題】**
もし `users_last_month` 側の `user_id` に1件でも `NULL` が含まれていた場合、`NOT IN` の論理評価は `UNKNOWN(NULL)` となり、**「結果が0件」になってしまう深刻なバグ**を抱えています。

---

## 3. ✅ 解決策1：EXCEPT (カラム全体での完全一致差分)

2つのクエリの結果同士を「集合的に引き算」する構文です。

```sql
-- 今月の全ユーザー集合 (A)
SELECT user_id, email FROM users_this_month
EXCEPT
-- 先月の全ユーザー集合 (B)
SELECT user_id, email FROM users_last_month

-- 結果 => (A - B) の差分。今月だけの新規ユーザーが出力される
```

**【メリットとデメリット】**
- **メリット**: まったく同じ構造のテーブル同士（スナップショット等）の比較に超強力。
- **デメリット**: SELECTした「すべての列」を比較して差分を出すため、「ID」だけをキーにして特定の状態やメールアドレスを引っ張るような複雑な条件式には不向きです。

---

## 4. ✅ 解決策2：LEFT ANTI JOIN (実務上の最強ソリューション)

最も実務的、かつオプティマイザが最高の効率（Hash Joinなど）を出せる結合方式です。
「左側のテーブルにはあるが、右側には**存在しない(マッチしなかった)**データのみ」を直接出力します。

```sql
SELECT 
  t.user_id, 
  t.email 
FROM 
  users_this_month t
-- ★ ANTI JOIN ＝ 一致しない行だけを残す特殊なJOIN
LEFT ANTI JOIN 
  users_last_month l 
  ON t.user_id = l.user_id;

-- ※ 従来の LEFT JOIN + IS NULL よりも意図が明確で高速です
```

## 5. 補足：既存の変更点を探したい場合 (INNER JOIN)

もし「IDは同じなんだけど、Emailアドレスが変わった人」を探したい場合はどうするか？
その場合は素直に `INNER JOIN` をして中身を比較します。

```sql
SELECT 
  t.user_id, 
  l.email as old_email,
  t.email as new_email
FROM users_this_month t
INNER JOIN users_last_month l 
  ON t.user_id = l.user_id
-- 旧データと新データで値が違う人
WHERE t.email != l.email;
```

## 6. 確認クイズ（理解度チェック）

### Q1. NOT IN と NOT EXISTS では、どちらが安全でパフォーマンスが良いでしょうか？

<details>
<summary>💡 回答を見る</summary>

**正解: `NOT EXISTS` です。**

`NOT IN` はサブクエリ内に一つでも NULL が含まれると一切機能しなくなる致命的弱点がありますが、`NOT EXISTS` は安全です。
ただし、現在のSparkエンジン等の基盤では `NOT EXISTS` と書いた瞬間に、内部で自動的に `LEFT ANTI JOIN` の物理プランに翻訳して実行してくれます。
</details>

### Q2. 「今月の退会者」を知るには上記の LEFT ANTI JOIN をどう書き換えればよいでしょうか？

<details>
<summary>💡 回答を見る</summary>

**正解: 左と右のテーブル指定を逆にします。**

先月のリスト(Left)にあって、今月のリスト(Right)に無い人を探せば良いので：
```sql
SELECT l.user_id 
FROM users_last_month l 
LEFT ANTI JOIN users_this_month t ON l.user_id = t.user_id
```
となります。
</details>
