# Time Series & Windows

> [!TIP]
> **「直近30日の移動平均を、日付別に出してほしい」**
> このオーダーに対し、日付の自己結合（Cross Join + 範囲WHERE）を書いてしまうとあっという間にシステムがメモリ不足でダウンします。
> データに触る回数を最小限に抑え、劇的にパフォーマンスを改善する「スライディングウィンドウ」の極意を解説します。

---

## 1. ビジネスシナリオ：トレンドの可視化

「日々の小売店の売上（daily_sales）が曜日によって乱高下するので、トレンドが見えない。過去7日間の移動平均（7-Day Moving Average）を算出してKPIダッシュボードに出したい。」

---

## 2. ❌ アンチパターン：非効率な自己結合

昔ながらのSQLでこれを書こうとすると、「集計基準日」に対して「過去7日分のデータ」を無理やり掛け合わせる形になります。

```sql
-- ❌ データ件数が数万行を超えたあたりから終わらなくなるクエリ
SELECT 
  t1.date,
  t1.daily_sales,
  AVG(t2.daily_sales) AS rolling_7d_avg
FROM retail_sales t1
-- 各々の日に、自分を含めた過去6日分を強引に結びつける
INNER JOIN retail_sales t2 
  ON t2.date BETWEEN DATE_SUB(t1.date, 6) AND t1.date
GROUP BY t1.date, t1.daily_sales
ORDER BY t1.date;
```
1年のデータ（365日）ごとに7回JOINされ、行数が爆発します。最悪のO(N^2)処理の一種です。

---

## 3. ✅ ベストプラクティス：ROWS BETWEEN

`OVER` 句の中に `ROWS BETWEEN` を指定し、「計算対象とする窓（スライディングウィンドウ）」を定義します。これにより、エンジンはデータを日付順に1回だけスキャンしながら、並行して平均を足し引き（ローリング）していく極めて効率的な計算（O(N)）を行います。

```sql
SELECT
  date,
  daily_sales,
  
  -- ★ 直前6行 ＋ 当日行 ＝ 計7行 の平均をとる
  AVG(daily_sales) OVER (
    ORDER BY date ASC
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS rolling_7d_avg
  
FROM
  retail_sales
ORDER BY date ASC;
```

---

## 4. ⚠️ 罠：カレンダーの抜け漏れ (ROWS vs RANGE)

もし「店が定休日で、売上レコードが存在しない日」があったらどうなるでしょう？

**【ROWS の場合】**
`ROWS BETWEEN 6 PRECEDING` は「純粋に上の行え数えて6行」です。
定休日があった場合、実際は「カレンダー上で10日前」のデータを拾ってしまい、巨大なスパイクがあった場合に平均を歪めます。

**【RANGE の場合】**
行数ではなく、「数値や日付の値」そのものを基準に窓を作ります。日付型の RANGE 計算をフルサポートしている環境（Databricks / Spark SQLなど）であれば、こちらが完全な正解です。

```sql
SELECT
  date,
  daily_sales,
  
  -- ★ 行数に依存せず、絶対的な「日付ベースで過去6日前まで」を見る
  AVG(daily_sales) OVER (
    ORDER BY CAST(date AS TIMESTAMP) 
    RANGE BETWEEN INTERVAL 6 DAYS PRECEDING AND CURRENT ROW
  ) AS exact_7d_avg
FROM
  retail_sales;
```

## 5. 確認クイズ（理解度チェック）

### Q1. 「累積売上高（YTD: Year to Date）」を出す場合、Windowの開始地点はどう記載するか？

<details>
<summary>💡 回答を見る</summary>

「その年の最初の行（どこまでも前）」から「現在」までなので、`UNBOUNDED PRECEDING` を使用します。

```sql
SUM(daily_sales) OVER (
  PARTITION BY year(date)
  ORDER BY date ASC
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
)
```
</details>

### Q2. 過去3日間の平均を出すため `ROWS BETWEEN 2 PRECEDING AND CURRENT ROW` を使いました。最初の2日分（対象が3日に満たない日）の計算はどう処理されますか？エラーになりますか？

<details>
<summary>💡 回答を見る</summary>

**正解: エラーにならず、存在する行だけで平均が計算されます。**

1日目はその日だけの平均、2日目は2日分での平均になります。
これを防いで「3日揃っていない時はNULLにしたい」場合は、`CASE WHEN COUNT(*) OVER (...) = 3 THEN ... END` のように事前にカウントチェックを噛ませるテクニックが必要です。
</details>
