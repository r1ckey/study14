# アウトプット逆算型の分析クエリ設計手法

> [!IMPORTANT]
> **「複雑なSQLを書く時、いきなり SELECT から書き始めていませんか？」**
> 分析用に数千行のコードを書く際、インプット（元データ）から順次結合していくアプローチは、パフォーマンス劣化やメンテナンス不能な「スパゲティコード」を生み出します。
> プロフェッショナルなデータエンジニアリングの基本である「アウトプットから逆算する設計」を完全にマスターし、チーム全員が読める堅牢なSQLを設計しましょう。

---

## 1. 原則：なぜ「インプット」から考えてはいけないのか？

データエンジニアがよく陥る罠が、**「テーブルAとテーブルBをとりあえずJOINして、そこから必要な列を絞り込む」**という思考プロセスです。

### 1.1 ❌ インプット駆動のアンチパターン（スパゲティコードの典型）
```sql
-- ❌ 思考停止でJOINを繋げた例（パフォーマンスが悪く、意図が不明）
SELECT 
    l.user_id,
    p.name,
    p.age_group,
    p.gender,
    c.campaign_name,
    SUM(l.view_time) as total_view_time,
    COUNT(DISTINCT l.video_id) as unique_videos
FROM view_logs l
LEFT JOIN user_profiles p ON l.user_id = p.user_id
LEFT JOIN campaigns c ON l.campaign_id = c.campaign_id
WHERE l.date BETWEEN '2025-01-01' AND '2025-01-31'
  AND p.status = 'active'
GROUP BY 
    l.user_id, p.name, p.age_group, p.gender, c.campaign_name;
```

**【このコードの何が悪いのか？】**
1. **意図が読めない**: 最終的に「何を集計したいのか」が最後まで読まないと分からない。
2. **無駄なリソース消費**: JOINしてからGROUP BYしているため、数十億行のログデータに対して巨大なシャッフル（Shuffle）が発生し、Sparkクラスターのメモリを圧迫します。
3. **デバッグの困難さ**: もし `total_view_time` の数値がおかしくても、どこで行が爆発したのか切り分けられません。

---

## 2. 解決策：アウトプット逆算思考

実務におけるプロのスタイルでは、**「最終的にBIツールでどう見せたいか（Output）」**から逆算して、段階的に `WITH` 句（CTE）を構築します。

### 2.1 思考プロセスの3ステップ
1. **Output（最終結果）**: まず、最終的に出力したいカラム構成を定義する。
2. **Process（集計・加工）**: その結果を得るために、各ファクト（履歴）テーブルを最適な粒度で事前集計（Pre-aggregation）する。
3. **Input（抽出）**: 必要なディメンション（マスタ）テーブルを準備する。

## 3. 実践：CTEによるモジュール化設計

アンチパターンで示したSQLを、**「アウトプット逆算＋事前集計」**で美しく書き直します。
各CTE（ブロック）は「1つの明確な目的」を持ち、後続から使い回せるように設計します。

### ✅ ベストプラクティスの実装

```sql
WITH 
-- ==========================================
-- 1. Input: 対象ユーザーの抽出（要件定義ブロック）
-- ==========================================
target_users AS (
    SELECT 
        user_id,
        name,
        age_group,
        gender
    FROM user_profiles 
    WHERE status = 'active'
),

-- ==========================================
-- 2. Process: トランザクションの事前集計（Pre-aggregation）
-- ※ JOINする【前】に粒度を圧縮し、シャッフルを最小化する
-- ==========================================
aggregated_views AS (
    SELECT 
        user_id,
        campaign_id,
        SUM(view_time) AS total_view_time,
        COUNT(DISTINCT video_id) AS unique_videos
    FROM view_logs
    WHERE date BETWEEN '2025-01-01' AND '2025-01-31'
    GROUP BY user_id, campaign_id
)

-- ==========================================
-- 3. Output: 結合と最終成形（Final Assembly）
-- ==========================================
SELECT 
    u.user_id,
    u.name,
    u.age_group,
    u.gender,
    COALESCE(c.campaign_name, 'No Campaign') AS campaign_name,
    v.total_view_time,
    v.unique_videos
FROM target_users u
-- 既に圧縮された小さなデータ同士をJOINするため超高速
INNER JOIN aggregated_views v 
    ON u.user_id = v.user_id
LEFT JOIN campaigns c 
    ON v.campaign_id = c.campaign_id;
```

> [!TIP]
> **パフォーマンス上の圧倒的メリット（Pre-aggregation）**
> 元の巨大なログテーブル（数十億行）を他のマスタと結合（JOIN）するのではなく、先に `aggregated_views` CTEの中で `GROUP BY user_id` を行い、数百万行レベルに**圧縮してから結合**しています。これにより、Sparkのシャッフルデータ量が1/100以下になり、OOMエラーを防ぐことができます。

## 4. なぜCTE（WITH句）で分割するのか？

実務においてCTEでクエリを細かく分割することは、単なる「見た目の問題」ではありません。**チーム開発におけるインフラ**です。

### 4.1 デバッグの容易性
例えば「数値がおかしい」と報告された場合、インプット駆動のクエリでは原因特定が困難です。
しかしCTEで分割されていれば、以下のように一部をコメントアウトして即座に検証（ユニットテスト）が可能です。

```sql
-- 最終抽出部をコメントアウトし、特定のモジュールだけを出力してみる
-- SELECT ... FROM target_users ...

SELECT * FROM aggregated_views LIMIT 100;
```
これにより「集計ロジックが間違っているのか」「結合キーが間違っているのか」を一瞬で切り分けることができます。

### 4.2 Catalyst Optimizer（Spark）との親和性
「CTEを使いすぎるとパフォーマンスが落ちるのでは？」と心配するエンジニアがいますが、**杞憂です**。
Databricksに搭載されている **Catalyst Optimizer** は記述された論理SQL（Logical Plan）を展開し、最も効率の良い物理実行計画（Physical Plan）に自動で並び替えます。
人間目線で「最も読みやすい粒度」で細かくCTEを切るのが正解です。

## 5. 確認クイズ（理解度チェック）

### Q1. 大量データのログテーブルとマスタテーブルを結合する際、JOINとGROUP BYはどちらを先に行うべきか？

<details>
<summary>💡 回答を見る</summary>

**正解: GROUP BY（集計）を先に行うのがベストプラクティス**

結合（JOIN）の前にデータを集計（Pre-aggregation）して行数を圧縮することで、高コストなシャッフル処理を劇的に削減できます。

```sql
SELECT c.name, COUNT(o.id) 
FROM customers c JOIN orders o ON c.id = o.customer_id 
WHERE o.status = 'shipped' AND c.region = 'Tokyo'
```

```sql
WITH tokyo_customers AS (...),
shipped_orders_by_user AS (...)

```
