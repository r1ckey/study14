# Logical vs Physical Design & Testing

> [!TIP]
> **「数百行のSQLを書いて、一発で本番環境に投入していませんか？」**
> 分析クエリもソフトウェア（コード）の一部です。自己修復機能・自己テスト機能を持たないコードは、データの変更や要件追加によって確実に壊れます。
> 本モジュールでは、一時表（TEMP VIEW）の罠と、CTEを利用した「自己テスト機能付きクエリ」のアーキテクチャを解説します。

---

## 1. 概念：論理設計（人が読む）と物理設計（マシンが読む）

データ分析基盤において、SQLエンジンの進化（特にSpark Catalyst Optimizer等のモダンなCBO）により、人間が書いたコードがそのまま実行されることはなくなりました。

### 1.1 論理プラン（Logical Plan）
我々人間が書くSQLそのものです。理解しやすさ、チームでの保守性、レビューのしやすさが100%優先されます。
無駄に見えるCTEの分割も、「人間が読むための論理的境界」として極めて重要です。

### 1.2 物理実行プラン（Physical Plan）
エンジンが背後で書き換える最適な実行手段です。「どのJOIN方式（Broadcast/SortMerge）を使うか」「どこまで先読み（Predicate Pushdown）するか」を決定します。

\`\`\`mermaid

    subgraph 人間が見る論理構造
    A[CTE 1: Users] --> B[CTE 2: Filter] --> C[CTE 3: Join]
    end

    D[最適化された 1つのプロセス<br>Shuffleの最小化]
    end
    
    C -.->|コンパイル&最適化| D
\`\`\`

**結論:** 「パフォーマンスが落ちそうだから」という理由で、無理に1つの巨大な `SELECT` に押し込んだり、ネストを深くする（サブクエリの嵐）必要はありません。CTEでふんだんに分割し、保守性を高めるべきです。

## 2. 実務の罠：TEMP表（一時表）はオプティマイザの敵

CTEで分割する代わりに、`CREATE TEMP VIEW` を使って中間データを保存するエンジニアがいます。

### 2.1 ❌ TEMP表が引き起こすパフォーマンス劣化
```sql
-- ❌ Step 1: ユーザーを絞る
CREATE TEMP VIEW tmp_active_users AS 
SELECT * FROM users WHERE status = 'active';

-- ❌ Step 2: ログとJOINする
SELECT a.*, l.log_time 
FROM tmp_active_users a 
JOIN logs l ON a.id = l.user_id 
WHERE l.log_time > '2025-01-01';
```

**【なぜ悪いのか？】**
1. **限界（Boundary）の発生**: Sparkのオプティマイザは、`tmp_active_users` の作成時点では「後で日付フィルタが掛かること」を知りません。
2. **プッシュダウン阻害**: 本来なら先にログを日付で絞り、超軽量化した状態でユーザーと結合すべきですが、TEMP表を間に挟むと「全体最適化の連鎖」が途切れます。

### 2.2 ✅ CTEによる全体最適化への委譲
```sql
WITH active_users AS (
    SELECT * FROM users WHERE status = 'active'
),
joined_logs AS (
    SELECT a.*, l.log_time 
    FROM active_users a 
    JOIN logs l ON a.id = l.user_id 
    WHERE l.log_time > '2025-01-01'
)
SELECT * FROM joined_logs;
```
1つのスクリプトとして完結しているため、オプティマイザが「下流の WHERE 条件」を上流まで引き上げ（Predicate Pushdown）、不要なデータの読み込みを完全にスキップします。

## 3. 実践：テスト用CTEの組み込み（Test-Driven SQL）

実務において「データが入っていない（NULL）」「負の売上が混ざっている」といった異常値は日常茶飯事です。

### 3.1 開発時の安全弁となる `test__` ロック

最終的な `SELECT` の前に、意図的に `test__` から始まるブロックを作成し、ユニットテストを組み込みます。

```sql
WITH 
  -- 1. 実データ処理ブロック
  base_data AS (
    SELECT * FROM transactions WHERE amount > 0
  ),
  aggregated AS (
    SELECT user_id, SUM(amount) AS total FROM base_data GROUP BY 1
  ),
  
  -- ★ 2. テスト用ブロック（ここが肝）
  test__check_null AS (
    -- user_idがNULLになっているやばいデータがないか検知
    SELECT user_id, total, 'ERROR: userId is NULL' as error_msg
    FROM aggregated 
    WHERE user_id IS NULL
  ),
  test__check_negative AS (
    -- 集計結果がゼロやマイナスになるバグがないか検知
    SELECT user_id, total, 'ERROR: Total is <= 0' as error_msg 
    FROM aggregated 
    WHERE total <= 0
  )

-- ==========================================
-- 【通常運用時】
SELECT * FROM aggregated;

-- 【開発中・バグ調査時】
-- 上記をコメントアウトし、テスト用CTEを呼び出す
-- SELECT * FROM test__check_null;
-- ==========================================
```

### 3.2 なぜこれが実務で強力なのか？
データのバグ調査を依頼された際、「どこがおかしいのか1からSQLを読み解く」のは時間の無駄です。
このようにテスト群がコード本体に埋め込まれていれば、他のエンジニアでも「テストブロックを順番にSELECTしてみるだけ」で不整合データの存在を瞬時に特定できます。

## 4. 確認クイズ（理解度チェック）

### Q1. TEMP VIEWを使うべきシーンとは？
<details>
<summary>💡 回答を見る</summary>

**正解: 異なる複数の言語（例えばPythonシェルからSQLの結果を使い回す場合）や、完全に別々のノートブック間でデータを一時的に共有したい場合。**

同一スクリプト・クエリ内で完結する処理であれば、最適化の観点から常に CTE（WITH句） を使うべきです。
</details>

### Q2. Predicate Pushdown（条件のプッシュダウン）とは？
<details>
<summary>💡 回答を見る</summary>

**正解: クエリの最後に書かれたWHERE句などの絞り込み条件を、システムが自動的に「データをストレージから読み込む一番最初の段階（下層）」まで押し下げてくれる最適化機能のこと。**

これにより、不必要なデータをメモリに乗せることなく、数TBのファイルから数十MBだけを読むような高速化が実現します。
</details>
