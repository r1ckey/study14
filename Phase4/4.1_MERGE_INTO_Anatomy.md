# MERGE INTO 解体新書：堅牢なデータパイプラインの最強の切り札

> [!CAUTION]
> **「バッチ処理の途中でエラーが起きた」**
> この時、すでにINSERTされてしまったデータを手動で消して再実行していませんか？
> \`MERGE INTO\` をマスターすれば、冪等性（何度実行しても必ず同じ結果になる）が担保され、エラーに強い堅牢なパイプラインが完成します。

---

## 1. そもそも MERGE INTO とは何か？

データ連携処理において、**「ターゲットにデータが既に存在していれば更新（UPDATE）、存在していなければ新規追加（INSERT）」**というロジックは実務で1日100回登場します。これを **Upsert（アップサート）** と呼びます。

### 1.1 ❌ 従来の危険なアプローチ
昔のデータベースや一部の処理では、これを2段構えで書いていました。

```sql
-- ❌ Step 1: まずUPDATEを試みる
UPDATE target_table t
SET t.value = s.value
FROM source_table s WHERE t.id = s.id;

-- ❌ Step 2: 次に存在しないものだけINSERTする
INSERT INTO target_table (id, value)
SELECT id, value FROM source_table s
WHERE NOT EXISTS (SELECT 1 FROM target_table t WHERE t.id = s.id);
```

**【何が危険なのか？】**
Step 1 と Step 2 の間でサーバーがクラッシュした場合、データが不整合を起こします。また、コードが二重になりメンテナンスビリティも最悪です。

### 1.2 ✅ MERGE INTO による解決
\`MERGE INTO\`を使えば、上記の処理を**「1つのACIDトランザクション」**として完結させることができます。落ちてもロールバックされるため安全です。

## 2. MERGE INTO 構文の解剖（実務テンプレート）

Databricks等における実務利用を想定し、最も堅牢な `MERGE INTO` の書き方を解説します。

\`\`\`mermaid
graph TD
    classDef target fill:#0f172a,stroke:#3b82f6,stroke-width:2px,color:#fff;
    classDef source fill:#0f172a,stroke:#10b981,stroke-width:2px,color:#fff;
    classDef match fill:#f59e0b,stroke:#fff,stroke-width:2px,color:#fff;

    A[Target Table<br>ターゲット]:::target
    B[Source Data<br>新しい差分データ]:::source

    A & B --> C{ON 句<br>IDの一致判定}

    C -->|一致した (WHEN MATCHED)| D[UPDATE / 必要な行だけ更新]:::match
    C -->|一致しない (WHEN NOT MATCHED)| E[INSERT / 新規レコード追加]:::match
\`\`\`

```sql
-- ① 【どこを】書き換えるか（ターゲット宣言）
MERGE INTO silver.customer_master AS target

-- ② 【どのデータを】持ち込むか（ソースのインライン定義）
-- ※ CTEを外で作るのではなく、USING句の中に直接書く事も可能
USING (
  SELECT 
    id, 
    name, 
    status,
    ROW_NUMBER() OVER(PARTITION BY id ORDER BY updated_at DESC) as rn
  FROM bronze.daily_customer_extract
  WHERE is_valid = true
) AS source

-- ③ 【何をもって】同一と判定するか（キー照合）
ON target.id = source.id 
   AND source.rn = 1  -- 注意：ソース側でIDが重複しているとエラーになるため1行に絞る

-- ④ 一致した場合の処理（UPDATE）
WHEN MATCHED 
  -- ★プロの技: 不必要な更新（無駄な書き込みIO）を防ぐため、値が変化した時だけ更新する
  AND (target.name != source.name OR target.status != source.status) 
THEN 
  UPDATE SET 
    target.name = source.name,
    target.status = source.status,
    target.updated_at = current_timestamp()

-- ⑤ 一致しなかった場合の処理（INSERT）
WHEN NOT MATCHED THEN
  INSERT (
    id, 
    name, 
    status, 
    created_at, 
    updated_at
  )
  VALUES (
    source.id, 
    source.name, 
    source.status, 
    current_timestamp(), 
    current_timestamp()
  );
```

## 3. 実務で「絶対に」やってはいけないアンチパターン

### 3.1 ❌ ソース側のユニーク制約違反
`MERGE` を実行した際、Target（既存）の1行に対して、Source（新）の**複数行**が一致してしまうと、エンジンは「どちらの値で更新すればいいか分からない」ため、深刻なエラーを吐いて停止します。

- **解決策**: `USING` 句の中で `ROW_NUMBER() = 1` 等を用いて、必ずSource側が**ユニーク（一意）**になるように前処理を行ってください。（フェーズ2参照）

### 3.2 ❌ 全カラムの盲目的なUPDATE
`WHEN MATCHED THEN UPDATE SET *` と書くと、値が変わっていなくても全行が更新対象となり、膨大な無駄な書き込み（トランザクションログの肥大化）が発生します。
- **解決策**: 先述のテンプレートのように `AND (target.val != source.val)` のような安全弁を必ず入れましょう。

## 4. 応用：論理削除（Soft Delete）との連動

「データが送られてこなくなった = 削除されたとみなす」という実務要件のケースです。
`WHEN NOT MATCHED BY SOURCE` という拡張構文（Databricks独自等）を利用します。

```sql
MERGE INTO silver.customer_master AS target
USING active_customers_from_api AS source
ON target.id = source.id

WHEN MATCHED THEN 
  UPDATE SET target.name = source.name

-- 「ターゲット(既存)には存在するが、ソース(最新API)には存在しなかった」場合の処理
WHEN NOT MATCHED BY SOURCE AND target.status != 'deleted' THEN
  UPDATE SET 
    target.status = 'deleted',
    target.deleted_at = current_timestamp()
```

## 5. 確認クイズ（理解度チェック）

### Q1. Source側のデータに全く同じIDが2行（鈴木さんと佐藤さん）含まれていて、Target側にそのIDが存在する場合、MERGE文はどうなるか？

<details>
<summary>💡 回答を見る</summary>

**正解: ランタイムエラー（実行時エラー）が発生する。**

どちらの値（鈴木か佐藤か）で更新すれば良いかエンジンが判断できないため、`MERGE` は失敗します。これを防ぐため、事前に一意化（重複排除）を確実に行う必要があります。
</details>

### Q2. 冪等性（べきとうせい）とはどういう意味ですか？MERGEがなぜそれを担保できるのですか？

<details>
<summary>💡 回答を見る</summary>

**正解: 「何度同じ処理を実行しても、結果が同じ状態になること」**

単なる `INSERT` の場合、2回実行するとデータが2重に（ダブって）入ってしまいます。しかし `MERGE` は既存キーを確認するため、何度実行しても「最新の状態が正しく反映されるだけ」となり、リトライ処理が極めて安全になります。
</details>
