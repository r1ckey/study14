# 4.1: Upsert処理の決定版（MERGE INTO）

---

### 1. 【エンジニアの定義】Professional Definition

> **Upsert (Update + Insert)**:
> レコードが存在すれば更新（Update）し、存在しなければ挿入（Insert）する操作。
>
> **MERGE INTO**:
> 2つのテーブルを特定の条件（キー）で突合し、一致した場合と一致しなかった場合の複数のアクション（UPDATE, INSERT, DELETE）を 1つのトランザクション内で実行するSQL構文。Delta Lakeの核となる機能。

---

### 2. 【0ベース・深掘り解説】Gap Filling

#### 🧐 なぜ UPDATE ＋ INSERT ではダメなのか？
1. `UPDATE ...` を実行
2. その直後に `INSERT ...` を実行

この 2ステップ方式だと、1 と 2 の間にエラーが発生した場合、データベースの中身が「更新はされたが追加はされていない」という不整合な状態になります。
また、逐次処理は非常に低速です。 `MERGE INTO` はこれらを 1つの「アトミック（不可分）」な処理として実行するため、**冪等性（Idempotency）**が保証され、再実行が安全になります。

### 4. 【技術実装】Implementation Best Practices

#### ✅ MERGE INTO の解体新書
```sql
-- ① 書き換え対象の「ターゲット」
MERGE INTO silver.customer_master AS target

-- ② 持ち込む「ソース」
USING (
  SELECT id, name, status FROM bronze.raw_daily_customers
) AS source

-- ③ 何をもって同一とみなすか（マージキー）
ON target.id = source.id

-- ④ 一致した場合：最新情報に更新
WHEN MATCHED 
  AND (target.name != source.name OR target.status != source.status) THEN 
  -- ※値が変わった時だけ更新するのが、ログやコスト削減のコツ
  UPDATE SET 
    target.name = source.name,
    target.status = source.status,
    target.updated_at = current_timestamp()

-- ⑤ 一致しなかった場合：新規追加
WHEN NOT MATCHED THEN
  INSERT (id, name, status, created_at, updated_at)
  VALUES (source.id, source.name, source.status, current_timestamp(), current_timestamp());
```

---

### 5. 【Key Takeaways】

- **安全な再実行**: ジョブが途中で落ちても、 `MERGE` なら何度実行しても最終的な状態は同じになる。
- **パフォーマンスの最適化**: ソースデータを絞り込んでから `USING` に渡し、 `MATCHED` 句にも変更検知条件（AND target.x != source.x）を入れることで、不必要な書き込みを避ける。
- **ACIDの遵守**: トランザクションとして処理されるため、ハーフコミット（中途半端な更新）を防げる。
