# SCD Type 2 Implementation

> [!IMPORTANT]
> **「なぜ過去の売上実績が勝手に変わっているんだ！？」**
> 分析基盤において「ユーザーの属性情報を単純上書き（UPDATE）」することは絶対のタブーです。
> 顧客の会員ランクは「通常」から「プレミアム」に変動します。売上分析は「当時のランク」で行わなければなりません。
> この履歴管理の課題を解決する、DWH設計の真髄『SCD (Slowly Changing Dimension) Type 2』を解説します。

---

## 1. 概念：SCD (ゆっくり変化する次元)

マスタデータ（次元テーブルの属性）の変更をどう扱うかには、世界共通の型（Type）が存在します。

### Type 1: 単純な上書き（Overwrite）
歴史的経緯はどうでもよく、現在の値だけを知りたい場合に使います。
住所変更など。過去の出荷データなども最新住所で集計されてしまうデメリットがあります。

### Type 2: 新規行の追加による履歴管理（Versioning）
データを上書きせず、「有効開始日（valid_from）」と「有効終了日（valid_to）」、さらに「最新フラグ（is_current）」を持たせることで履歴の断面を残します。

**【顧客次元（dim_customer）の例】**

| ID | ランク | valid_from | valid_to | is_current |
|---|---|---|---|---|
| 01 | 通常 | 2022-01-01 | 2023-06-01 | FALSE |
| 01 | プレミアム | 2023-06-01 | 9999-12-31 | **TRUE** |

## 2. ❌ 自作SQLによる泥臭い実装（MERGEの応用）

SCD Type 2の実装は、「現在の有効な行を古くする（無効化 UPDATE）」しつつ、「新しい行を追加（INSERT）」するという2つのアクションが必要になります。純粋な `MERGE INTO` だけで書こうとすると以下のような黒魔術になります。

```sql
-- ❌ エラーを踏みやすく、可読性も低いアンチパターン
-- (仕組みを知るための概念コードであり、実運用では避けるべき)
MERGE INTO dim_customer AS target
USING (
  -- ① 値が変わった更新用のフラグ行
  SELECT id, rank, 'UPDATE' as act FROM cdc_changes
  UNION ALL
  -- ② 新しい値の追加用のフラグ行
  SELECT id, rank, 'INSERT' as act FROM cdc_changes
) AS source
ON target.id = source.id AND target.is_current = true

-- 更新用データと一致した場合、既存レコードを「無効化」
WHEN MATCHED AND source.act = 'UPDATE' AND target.rank != source.rank THEN
  UPDATE SET 
    is_current = false,
    valid_to = current_timestamp()

-- 追加用データの場合は、新規レコードとして書き込み
WHEN NOT MATCHED THEN
  INSERT (id, rank, valid_from, valid_to, is_current)
  VALUES (source.id, source.rank, current_timestamp(), '9999-12-31', true)
```

この実装はトランザクション順序の考慮や、同日に2回変更が来た場合の考慮漏れ（CDCの順序性担保）が起きやすく、データエンジニアリング上の大きな技術的負債となります。

## 3. ✅ 現代的な解決策：フレームワークへの委譲（DLT APPLY CHANGES）

データパイプライン構築における最大のベストプラクティスは、「車輪の再発明をしない」ことです。
Databricksでは、**Delta Live Tables (DLT)** という機能の `APPLY CHANGES INTO` 命令を使うと、この複雑なSCD Type 2の履歴管理を**パラメータだけで完全自動化**してくれます。

### 3.1 DLT (Python) によるエレガントな実装

```python
# 魔法のようにType 2の履歴管理テーブルが完成する
dlt.apply_changes(
  target = "dim_customer_history",
  source = "customer_cdc_stream",
  keys = ["id"],
  # 変更イベントが発生した時間のカラム。処理の前後が入れ替わって到着しても保証される
  sequence_by = dlt.expr("update_timestamp"),
  
  # Type 2を明示的に指定（デフォルトはType 1）
  stored_as_scd_type = "2",
  
  # 削除フラグを持つカラムがあればそれも渡す
  apply_as_deletes = dlt.expr("is_deleted = true")
)
```

**【裏側で何が起きているか？】**
自動的に `__START_AT` や `__END_AT` というシステムカラムが追加され、裏側のSparkエンジンが最も安全でパフォーマンスの高い方法でUpsertや履歴保存の `MERGE` を実行し続けてくれます。

## 4. 確認クイズ（理解度チェック）

### Q1. ファクト（売上実績など）側から Type 2のディメンション（顧客マスタなど）をJOINする際、結合条件にはどのような工夫が必要ですか？

<details>
<summary>💡 回答を見る</summary>

**正解: IDの一致だけでなく、売上発生日が「マスタの有効期間内」に収まっているかを条件に入れます。**

```sql
SELECT f.sales_amount, d.rank
FROM fact_sales f
INNER JOIN dim_customer d
  ON f.customer_id = d.id
  -- ファクトの日付が、その当時のマスタの有効期間内であること
  AND f.sales_date >= d.valid_from 
  AND f.sales_date < d.valid_to
```
これによって「購入した当時の会員ランク」を正確に引っ張ることができます。
</details>

### Q2. SCD Type 3 とは何ですか？

<details>
<summary>💡 回答を見る</summary>

**正解: 横持ちで履歴を一つだけ持つタイプです。**
`old_rank` と `current_rank` のように、前の状態だけをカラムとして横に持ちます。直近一回の変化だけ見えればよい安価な管理手段ですが、最近の大容量DWH環境ではType 2に統一されることがほとんどです。
</details>
