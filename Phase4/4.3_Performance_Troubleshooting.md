# Performance Troubleshooting

> [!CAUTION]
> **「昨日まで終わっていたクエリが、今日はいつまで経っても終わらない」**
> **「SparkクラスターがOOM（OutOfMemory）で頻繁にクラッシュする」**
> エンジニアなら誰もが経験するこの現象に対し、盲目的に「サーバーのスペック（メモリ）を上げる」という対処をするのはプロ失格です。
> 実行計画（Execution Plan）から犯人を立証する、トラブルシューティングの極意を解説します。

---

## 1. 致命的な原因①：Cross Join (直積の嵐)

トラブルの8割はこれと言っても過言ではありません。

`JOIN` の `ON` 条件が抜けていたり、無意味な評価（`ON 1=1`）になっていたりすると、システムは表Aの全行と表Bの全行を「総当たり」で掛け合わせる **Cartesian Product（デカルト積）** を生成します。

**【爆発の計算式】**
- 1万行 × 1万行 ＝ 1億行
- 10万行 × 10万行 ＝ 100億行（即死レベル）
- 1000万行 × 1000万行 ＝ 宇宙の終わり

### 📍 解決策・予防策
- 意図的に行う特別な用途（1行の定数テーブルとのJOINなど）以外、**必ず意味のあるビジネスキーで `INNER JOIN` / `LEFT JOIN` をする**。
- `EXPLAIN` の中に `CartesianProduct` という単語が見えたら、即座にクエリを停止させてレビューし直すこと。

## 2. 致命的な原因②：Data Skew (データの偏り・スキュー)

「クラスター全体のリソースは空いているのに、ジョブがずっと99%で止まっている」。
これは、一見するとシステム側の不具合に見えますが、大抵は**Data Skew**が原因です。

**【Data Skewとは何か？】**
特定の `id` （例えば異常値である `NULL` や、デフォルト値 `"Web"`、大口顧客のIDといった、「極端に大量のレコードを持つ値」）で `GROUP BY` や `JOIN` を行うと発生します。

Sparkなどの分散処理エンジンは、データの種類（キーのHash値）ごとに各サーバー（CPUコア）へタスクを分散させます。

### 📍 解決策・予防策 (モダンなアプローチ)
もしあなたが Databricks のような最新環境を使っているなら、小手先のコード修正ではなく **AQE (Adaptive Query Execution)** に頼るのが最適です。

```sql
-- 【処方箋】AQEのスキュー最適化オプションをONにする（通常はデフォルトでON）
SET spark.sql.adaptive.enabled = true;
SET spark.sql.adaptive.skewJoin.enabled = true;
```
AQEは、実行途中に「特定のキーだけ異常にデカい」と悟ると、そのキーのためだけにタスクを自動分割して並列処理に切り替えてくれます。

## 3. パフォーマンス改善の小技集（チートシート）

インフラのアップグレードを打診する前に、以下の3つは必ず試してください。

### 3.1 事前にフィルタリング (Predicate Pushdownの補佐)
`JOIN` する対象が巨大なログテーブルの場合、結合してから `WHERE` で年月などを絞るのではなく、**事前にCTE等で対象年代だけのサブセットを作ってからJOINする**のが鉄則です。（Data Skippingの活用）

### 3.2 Broadcast JOIN の強制
結合する片方のテーブルが非常に小さい場合（数万行程度までのマスタなど）、ヒントを使用して強制的に全ノードにブロードキャスト（コピー配布）させます。これにより、高コストな「シャッフル通信」を完全に回避できます。

```sql
SELECT /*+ BROADCAST(dim_table) */ * 
FROM fact_table 
JOIN dim_table ON fact_table.item_id = dim_table.item_id
```

### 3.3 OPTIMIZE と ZORDER (物理配置の最適化)
頻繁に `WHERE` や `JOIN` に使われる列（日付やユーザーIDなど）がある場合、ストレージ上でその列が近くに集まるよう、物理的にデータを並べ替えておきます。
これにより、クエリ実行時に「関係ないファイルを開く」時間を劇的に短縮（数十倍の高速化）できます。

```sql
-- Delta LakeにおけるZORDER最適化
OPTIMIZE my_logs_table ZORDER BY (user_id, event_date);
```

### Q1. トラブルシューティングの際、クエリの先頭に `EXPLAIN FORMATTED` をつけると何が出力されますか？

### Q2. データの偏り（Skew）の原因になりやすい代表的な値は何ですか？

<details>
<summary>💡 回答を見る</summary>

**正解: `NULL`、またはシステム上のデフォルト値です**

特に大量のログが「未登録ユーザー（user_id = NULL）」で送られてきている場合、`GROUP BY user_id` や `JOIN` を行うと NULL が一つのCPUコアに全集中してスパイクします。
`WHERE user_id IS NOT NULL` で先に切り捨てるか、ダミーのランダムIDを振って分散させる（Salting手法）などの前処理が必要です。
</details>
